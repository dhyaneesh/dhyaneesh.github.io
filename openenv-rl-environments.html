What I Learnt Building RL Environments with OpenEnv

I built Pwned — a deterministic, partially observable cyber-range benchmark — on top of Meta-PyTorch’s OpenEnv. Here is what the architecture taught me about why HTTP-native environments are the right abstraction for post-training RL, and what it means for the decade-long reign of OpenAI Gym.

Dhyaneesh DS / April 2026 / ~2,600 words
01 /
The Gymnasium Problem

OpenAI Gym built the wrong abstraction for the wrong era

OpenAI Gym shipped in 2016 and did exactly what it was designed to do: give researchers a standard API for training agents on classic control tasks. CartPole, Atari, MuJoCo. The interface was elegant — reset() returns an observation, step(action) returns the next one plus a reward — and it unified a decade of RL research under a single import.

But Gym was built around a specific model: one process, one machine, observation spaces defined as NumPy arrays, environments imported as Python classes. That model was correct for the research it was designed to serve. It is the wrong model for what RL is used for now.

When you are fine-tuning a language model with GRPO, you are running a distributed training job across multiple GPUs, generating thousands of rollouts in parallel, and scoring them with a reward model that may itself be running on a separate host. The environment in this setting is not a local Python class. It is a service. Treating it like a local Python class is the source of a lot of accidental complexity that shows up as infrastructure glue code, not research.

Gym was an API for a Python process. OpenEnv is an API for a network service. That distinction matters more than it sounds.

The problems compound when you try to share environments. With Gym, sharing meant publishing a Python package, version pinning, and hoping the recipient had matching system dependencies. This made it nearly impossible to build a genuine ecosystem of training tasks — the equivalent of a dataset hub, but for environments.

02 /
What OpenEnv Actually Is

Three verbs, a Docker container, and a Hugging Face Space

OpenEnv is not a drop-in replacement for Gymnasium. It does not ship with Atari wrappers or continuous control benchmarks. What it ships with is a pattern: every environment is a FastAPI server running in a Docker container, exposing three endpoints — /reset, /step, and /state — over HTTP and WebSocket. The client is a typed Python class that speaks to that server from anywhere on the network.

The API shape is intentionally Gymnasium-adjacent. Anyone who has used Gym will read the quick-start in thirty seconds. But the mechanism is entirely different. The environment is not imported — it is connected to. The observation is not a NumPy array — it is a Pydantic model. The action is not a gym.Space sample — it is a typed dataclass with validation. And the server can run anywhere: your laptop, a cloud VM, or a Hugging Face Space, deployed in one command.

The three core verbs

reset() — Initialise a new episode. Returns a typed Observation and a session identifier. Accepts seed and episode_id for exact reproducibility across training runs.

step(action) — Advance the episode by one action. Returns a StepResult: observation, reward, done flag, and optional metadata. The action is a validated Pydantic model, not a raw array.

state() — Return episode state without advancing it. Useful for debugging, multi-client scenarios, and benchmark inspection without consuming a step budget.

The typed model is what most changes how you design environments. In Gym, your observation was whatever you returned from step() — usually a NumPy array, sometimes a dict. In OpenEnv, you define an Observation class with named Pydantic fields. That class becomes the contract between the environment and any RL framework that connects to it. The server validates before sending; the client validates before handing to the training loop. What sounds like overhead eliminates an entire class of bugs that otherwise appear as NaN losses three hours into a training run.

Building Pwned on OpenEnv

What designing a cyber-range benchmark taught me

Pwned is a deterministic, partially observable cyber-range built on OpenEnv. It has two public suites: a Pentesting suite where agents pivot through a hidden network topology using commands like nmap, ssh, and exfiltrate; and a SecOps suite where agents respond to incident tickets using evidence they must observe and cite. Both suites share the same HTTP transport, the same session model, and the same deterministic reward engine. Building on OpenEnv forced three design decisions I would not have thought to make starting from a Gym subclass.

The session model is not optional when you run concurrent episodes

OpenEnv’s reset() returns a session_id. Every subsequent call includes that ID. This is the correct default because HTTP is stateless — without a session identifier threaded through every request, you cannot run multiple episodes concurrently against the same server, which is exactly what distributed RL training does. In Gym, state was implicit — it lived in the environment object. Running two episodes simultaneously required threading locks or spawning separate processes. OpenEnv makes the session boundary explicit and puts it in the transport. That lets one server handle hundreds of concurrent episodes cleanly.

The public/private state boundary is an architectural guarantee, not a naming convention

Pwned has a hidden network topology — the true graph of hosts, subnets, and pivot paths is never sent to the agent. In a local Gym environment, hidden state and observed state live in the same Python object. Nothing stops you from accidentally logging the full state to training metrics, leaking information the agent should not have. OpenEnv’s client/server boundary enforces this at the protocol level: PwnedPublicState is what the server serialises. The hidden PwnedEnvironment.state stays in-process and is never written to the transport. Partial observability becomes an architectural guarantee, not a convention that a careless evaluation script can violate.

Evidence minting is a temporal contract, not a data structure

The SecOps suite requires agents to cite evidence by ID in their final report. Evidence entries are minted at observation time — the moment an agent receives a StepResult, the observable facts are stamped with deterministic IDs like ev-0001. They cannot be backfilled. An agent that cites ev-0023 when the episode only produced twelve entries loses score. This maps exactly onto OpenEnv’s step-based model. Each step() call returns one observation — the atomic unit of what the agent knows. Minting at observation time creates an auditable record of when each piece of information became available, making benchmark grading reproducible: same seed, same actions, same cited IDs produces the same score every time.

Why Post-Training RL Needed This

The HTTP latency trade-off only cuts one way

Classic RL objections to HTTP-based environments are real: an in-process env.step() call takes microseconds; an HTTP round-trip takes milliseconds. For Atari or continuous control tasks requiring millions of steps, that gap is prohibitive. Vectorised environments — 64 or 256 copies of a Gym environment running in parallel on one machine — exist precisely to amortise per-step compute and keep the GPU busy.

But post-training RL on language models has inverted this math. Each “step” in a language model episode is a forward pass through billions of parameters — hundreds of milliseconds to seconds. An HTTP round-trip to the environment is noise in that budget. The bottleneck is the model, not the transport. And unlike Atari, where you can vectorise 256 environments on one CPU, language model training already runs across multiple GPU nodes. The training process and the environment need to be on different hosts by default, not by exception.

RFC 004 — Delayed Rewards

OpenEnv RFC 004 adds trajectory-level scoring: the reward signal is computed over an entire episode rather than at each step. This is the native shape of GRPO — generate a rollout, score it holistically, compute the gradient from the outcome. Making this a first-class protocol feature means training frameworks do not need to hack around a step-level API to implement outcome-based reward.

The Hugging Face Spaces deployment model extends this further. An OpenEnv environment pushed to HF Spaces is publicly addressable at a stable URL. A TRL training script on a GPU cluster, an Oumi experiment on a university machine, a SkyRL job on a cloud VM — all of them can connect to the same environment without infrastructure coordination. The environment operator and the model trainer do not need to be the same person. That is a fundamentally different relationship than anything Gym could offer.

Gym vs OpenEnv

The architectural reckoning

The two models reflect different assumptions about what RL training looks like. Neither is universally better. But one of them matches how frontier model training is actually done.

  • Session-based HTTP — one server handles hundreds of concurrent episodes with no process forking
  • Deployed to HF Spaces — addressable by any training framework via a stable URL, no install required
  • RFC 004 delayed rewards — trajectory-level scoring is a first-class environment protocol feature
  • Transport enforces the boundary — hidden state cannot be serialised to the wire, partial obs guaranteed

OpenEnv does not win on step throughput for classic control. It wins on the constraints post-training RL actually has: distributed training, typed observations, shared addressability, and trajectory-level reward.

The Gym model had one advantage OpenEnv has not recovered: raw throughput. In-process calls are nanoseconds; HTTP is milliseconds. For classic RL tasks requiring millions of steps, that gap is prohibitive. OpenEnv is not trying to win where Gym is strong. It targets the regime where Gym’s performance assumptions no longer hold — and where the distributed, multi-party, service-oriented model is the only one that scales.

Democratisation

The environment ecosystem that should exist but does not yet

Datasets for language model training have a hub. You can search Hugging Face, preview a dataset, and load it in three lines of code. Training environments have no equivalent. You can publish a paper, link a GitHub repo, and hope someone manages to reproduce your setup. Or you can push to HF Spaces with openenv push and hand anyone a URL.

The barrier to contributing a training environment used to require understanding the MDP formalism, reading the Gym registration docs, testing compatibility with every RL framework, and maintaining a pip package. The OpenEnv model reduces this to: implement three methods, write a Dockerfile, run one command. Your environment is then accessible to TRL, SkyRL, ART, Oumi, and Unsloth training scripts without any additional integration work, because they all speak the same HTTP protocol.

This is how ecosystem flywheels start. The Hugging Face datasets hub works because contributing friction is near zero and discoverability value is high. OpenEnv has the same structure in principle. The catalog is nascent — a handful of toy environments, a chess env, some Atari wrappers — but the infrastructure bet is that if you make environments as shareable as datasets, the research surface area any single lab can explore expands dramatically. Instead of every team building and maintaining their own training environments, the field converges on a shared catalog and spends its compute budget on the models, not the infrastructure.

Lessons from Pwned

Four things I would tell myself before starting

01
Design the observation first
Your Observation class is your environment’s public API
In Gym you could change what your environment returned at any time because it was just an array. In OpenEnv, your Observation is a Pydantic model that training frameworks depend on. Changing it breaks the wire format. Spend time getting the field names, types, and nullability right before writing any simulation logic. The typed model is a feature, not a constraint — but only if you design it intentionally.
02
Session hygiene
Treat session_id as a first-class resource, not a routing token
Concurrent evaluation means multiple sessions running simultaneously. If your environment has any global mutable state — a random seed, a shared data structure, a file handle — it will cause races that are nearly impossible to debug under concurrent load. The session model is only useful if the environment behind it is genuinely session-scoped. Design for concurrency from the start; retrofitting it is painful.
03
Reward design
The step penalty is not cosmetic — it is your efficiency signal
Pwned charges −0.01 per step and −1.01 on detection or tamper. The step penalty punishes noisy, reckless action sequences that happen to arrive at the right answer. Without it, a policy that runs nmap exhaustively on every subnet will outscore one that reasons about what it has already discovered. If you want agents that are efficient as well as effective, the reward function needs to make efficiency legible from the start.
04
Benchmark vs training environment
Decide which one you are building and make it unambiguous
A training environment should maximise signal density: reward shaping, curriculum, dense feedback at every step. A benchmark should maximise reproducibility: deterministic grading, fixed rubrics, no stochastic reward. Pwned is a benchmark first — same seed, same actions, same cited evidence IDs produces the same score every run. This made the grader design straightforward but constrained the reward engineering. Know your primary user before you write a line of simulation logic.
08 /
The Bottom Line

Where this is going

OpenEnv is early — the team flags it explicitly, the API will change, and the environment catalog is thin compared to what Gymnasium built up over a decade. But the underlying architectural bet is correct: environments for post-training RL should be network services, typed, containerised, and deployable to shared infrastructure.

The research that needs to happen next — better reward functions, harder tasks, more realistic simulation — will be easier if the infrastructure for sharing environments is already settled. The analogous moment for datasets was when Hugging Face standardised the dataset loading API and launched the Hub. Before that, every paper had its own preprocessing script and download link. After that, reproducing a paper’s data pipeline was one line of code. OpenEnv is trying to do the same thing for training environments. Whether the catalog reaches critical mass is an open question. But the architecture is right, and the timing — coinciding with post-training RL becoming the dominant method for improving frontier models — is better than it looks from the outside.

The next step for RL research infrastructure is not better algorithms — it is better shared environments. OpenEnv is the right bet on what that looks like.
